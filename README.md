📋 JPXデータ自動取得システム 構築・運用手順書
手元のGitHub Codespaces（開発環境）で安全にテスト・デバッグを行い、GitHub Actions（本番環境）で毎日高速に自動実行させるための手順書です。

1. フォルダ・ファイル構成（リポジトリの正しい形）
リポジトリのトップ階層は、必ず以下の構造になっている必要があります。

📂 N225OP_GEX (リポジトリルート)

📂 .github / 📂 workflows / 📄 jpx_scheduler.yml （GitHub Actionsの自動実行設定ファイル）

📂 .vscode / 📄 launch.json （Codespacesでのデバッグ用構成ファイル）

📄 main.py （Pythonのデータ取得メインプログラム）

📄 requirements.txt （高速化・キャッシュ用ライブラリ一覧）

📄 README.md （本手順書・このファイル）

2. 自動実行の仕組みと高速化（キャッシュ）について
◆ GitHub Actionsの仕組み
毎日指定のスケジュール（日本時間 17:30）になると、クラウド上で自動的にクリーンな実行環境が立ち上がり、main.py を実行してデータをGoogleスプレッドシートへ書き込みます。

◆ キャッシュ機能による高速化
requirements.txt を配置し、Workflowファイル側でキャッシュを有効化に指定することで、毎回発生していたライブラリ（Pandasなど）のダウンロード・インストール時間をスキップできます。これにより、2回目以降の実行速度が数秒にまで大幅に短縮されます。

3. Codespacesでの超効率デバッグ手順
GitHub Actionsへプログラムを送信（Push）する前に、オンラインVS Code上でエラーを1行ずつ特定してテストする手順です。

ステップ1：開発環境へのライブラリインストール（初回のみ）

Codespacesの下部にある「ターミナル（Terminal）」タブを開き、ライブラリ一括インストールコマンドを実行して開発環境を整えます。一度行えば、次回以降は不要です。

ステップ2：Google認証情報のセット

Google Cloudからダウンロードした秘密鍵（JSON）のテキストをすべてコピーし、Codespacesのターミナル上で環境変数として一時的に登録します。値は必ずシングルクォーテーションで囲んで実行してください。

ステップ3：プログラムの一時停止（ブレークポイント）の設定

main.py の中で、挙動を確認したい行やエラーが疑われる行の手前に、直接一時停止用の関数（breakpoint）を1行書き込んでファイルを保存します。

ステップ4：デバッガーの起動と検証

main.py を開いた状態にします。

キーボードの F5キー を押してデバッグを開始します。

プログラムが起動し、ステップ3で設定した位置で自動的に黄色くハイライトされ、一時停止状態になります。

画面上部に出現するコントロールパネルの「ステップオーバー（F10）」ボタンを1回ずつ押すことで、コードを1行ずつ安全に進めながら、エラーの発生箇所とエラーメッセージを特定できます。

4. Gitの同期・マージエラーが出た時の対処法
手元のCodespacesとGitHubの間でコードの状態がズレてしまい、分岐のエラー（divergent branches）が出てPull/Pushができなくなった場合の復旧手順です。

◆ 基本の解決手順（ルールを決めて合体する）
ターミナルで、Gitに対して「履歴の組み換えを行わずにマージ（合体）する」という設定コマンドを登録します。

その後、再度GitHubから最新のコードを引っ張ってきます（git pull）。

※もしこの際、画面に英語のマージメッセージが自動で開いた場合は、何も書き換えずにそのファイルのタブを閉じるか、ターミナル上で終了命令（:wq ➔ Enter）を入力して画面を閉じれば合体が確定します。

最後に、合体した最新の状態をGitHubへ送信（git push）して同期を完了させます。

◆ 究極の解決手順（手元の変更を破棄して強制リセット）
コードが複雑に絡まりすぎて修正が困難になった場合は、手元の変更をすべて破棄し、GitHub上にある最新の正常な状態を強制的にダウンロードして手元の環境を完全に上書き・一致させることができます。
